# MSS Mobile App — v1.2 Auth Redesign

**Spec date:** 2026-05-13
**Status:** Brainstorm-approved, ready for implementation plan
**Replaces:** v1.1 magic-link-only sign-in on `/mobile/`
**Scope:** customer-facing PWA at `mss-site/mobile/` (wrapped as the live iOS app and the in-flight Android app), plus a hardening + visibility pass on the admin portal `/admin/` so Mandeep and the team can see and manage every new customer agent.

---

## 1. Why this change

v1.1 shipped a magic-link sign-in card on the Track tab. It works, but:

- The owner is not happy with magic-link as the primary login UX.
- Real-estate agents (the target audience) expect a conventional **email + password** account.
- Without a registered profile, every booking has to re-type name / email / phone / agency, which kills repeat-booking conversion.

This v1.2 redesign replaces magic-link sign-in with a proper register-and-sign-in flow, adds a `profiles` table, and pre-fills the booking form from the signed-in user's profile.

## 2. User model

Two distinct contexts, kept separate:

| Context | Who | How they auth | Built where |
|---|---|---|---|
| **Mobile app** (`/mobile/`) | Real-estate agents (customers) | Email + password | This spec, Phases 1–5 |
| **Admin portal** (`/admin/`) | Mandeep + team | Email + password, **gated by `admin_users` membership** | This spec, Phase 6 |

### 2.1 Critical security finding (must fix in v1.2)

`mss-site/admin/auth-supabase.js:47` hardcodes `role: 'admin'` on every successful Supabase sign-in. Because customer registrations and admin sign-ins share the same `auth.users` table, **any customer who registers via `/mobile/` can today sign into `/admin/login.html` with their customer credentials and be granted admin role**. This is latent today because no customers exist, but the moment v1.2 opens registration it becomes exploitable.

Phase 6 closes this gap by requiring admin sign-in to verify `auth.uid()` is present in the new `admin_users` table. Sessions that aren't in `admin_users` are signed back out immediately with a clear error.

## 3. Approval flow

**Instant access.** The pipeline is:

```
Register form → email verification link sent → user clicks link
  → app opens, user is auto-signed-in → lands on pre-filled Booking Form
```

No admin gate. Filtering happens at the booking-confirmation step inside `/admin/` (Mandeep manually reviews each booking).

## 4. Sign-up fields (all required)

| Field | Notes |
|---|---|
| Email | Login identity, gets verification link |
| Password | 8+ characters, Supabase default rules |
| Full name | Used on bookings and outbound emails |
| Agency name | e.g. "Ray White Geelong" — helps Mandeep spot known agencies |
| Mobile phone | AU format (`+61…` or `04…`) — for fast contact about bookings |

## 5. Auth approach — chosen

**Approach A: Pure Supabase Auth.** No new Vercel API functions.

- `supabase.auth.signUp({ email, password })` for register
- `supabase.auth.signInWithPassword(...)` for sign-in
- `supabase.auth.resetPasswordForEmail(...)` for forgot password
- Email verification handled by Supabase (already SMTP-configured for `modernspacestyling@gmail.com`)
- Extra fields stored in a `profiles` table linked 1:1 to `auth.users`

Rejected alternatives:

- **B. Supabase Auth + custom `/api/register`** — adds a Vercel function. We're at 11/12 on the Hobby ceiling. Reserved for if we later need custom welcome emails.
- **C. Custom auth from scratch** — rejected. Security risk with no benefit.

## 6. Data model

### 6.1 `profiles` table (new)

```sql
create table public.profiles (
  id            uuid primary key references auth.users(id) on delete cascade,
  full_name     text not null,
  agency_name   text not null,
  mobile_phone  text not null,
  email         text not null,
  created_at    timestamptz default now(),
  updated_at    timestamptz default now()
);

alter table profiles enable row level security;

create policy "profiles_self_read"
  on profiles for select using (auth.uid() = id);
create policy "profiles_self_update"
  on profiles for update using (auth.uid() = id);
create policy "profiles_self_insert"
  on profiles for insert with check (auth.uid() = id);
```

### 6.2 `bookings` table changes

```sql
alter table bookings add column user_id uuid references auth.users(id);
create index on bookings(user_id);

-- Backfill: link existing rows where agent_email matches a real auth.users.email
update bookings b
   set user_id = u.id
  from auth.users u
 where b.user_id is null
   and lower(b.agent_email) = lower(u.email);

create policy "bookings_own_select"
  on bookings for select using (auth.uid() = user_id);
```

Rows without a matching auth user keep `user_id = NULL` and remain visible only in `/admin/`.

INSERT and UPDATE on `bookings` are handled by `POST /api/create-booking` running with the Supabase service role key, which bypasses RLS. Client-side direct writes are not allowed — no INSERT/UPDATE policy is needed for the anon role.

### 6.3 `admin_users` table (new, Phase 6)

```sql
create table public.admin_users (
  id          uuid primary key references auth.users(id) on delete cascade,
  email       text not null unique,
  full_name   text not null,
  added_by    uuid references auth.users(id),
  created_at  timestamptz default now()
);

alter table admin_users enable row level security;

-- An admin can read the admin_users table to see the team and verify membership
create policy "admins_read_team"
  on admin_users for select
  using (exists (select 1 from admin_users where id = auth.uid()));

-- Only admins can add new admins (via team.html, which calls a service-role API)
-- Inserts are gated server-side; no client-side INSERT policy is granted.

-- Seed Mandeep as the first admin
insert into admin_users (id, email, full_name)
  select id, email, 'Mandeep Singh'
    from auth.users
   where email = 'modernspacestyling@gmail.com';
```

Booking and profile RLS policies are extended to also allow admins to read any row:

```sql
create policy "profiles_admin_read"
  on profiles for select
  using (exists (select 1 from admin_users where id = auth.uid()));

create policy "bookings_admin_read"
  on bookings for select
  using (exists (select 1 from admin_users where id = auth.uid()));
```

## 10. Implementation phases

Five phases, each its own commit and Vercel deploy. iOS app picks up changes automatically (it wraps the live site via WKWebView).

| # | Phase | Ships | Gate to next phase |
|---|---|---|---|
| 1 | DB foundation | `profiles` table + RLS, `bookings.user_id` column + RLS, backfill | SQL runs clean in Supabase, RLS denies cross-user reads (verified by automated test) |
| 2 | Auth screens | Sign-In, Register, Forgot Password, Verify Email pages — all Supabase JS SDK, no new API routes | Fresh test user registers end-to-end, lands on verify-email screen |
| 3 | Profile + pre-fill | `profiles` row inserted after `auth.signUp` returns; Booking Form reads profile and pre-fills | New user lands on Booking Form with all 4 personal fields pre-filled |
| 4 | Booking + My Bookings gates | Booking form and My Bookings require auth; new booking writes `user_id`; My Bookings scoped via RLS | Test user submits booking, sees it in My Bookings within 5 seconds |
| 5 | Magic-link removal + existing user migration | Magic-link UI card removed from `/mobile/`; existing user resets password; verify-on-iOS-device pass | Mandeep signs in with email + new password on iOS, sees `MSS-2026-0011` |
| 6 | **Admin hardening + visibility** | `admin_users` table seeded; `/admin/auth-supabase.js` gates sign-in on `admin_users` membership (non-admins get a clear error and are signed out); `/admin/agents.html` lists every `profiles` row with search + sort; `/admin/bookings.html` joins `profiles` to show the owning agent for each booking; `/admin/dashboard.html` adds "Registered agents" + "New this week" stats; new `/admin/team.html` lets Mandeep invite team members by email (Supabase invite flow). Phase 6 also extends `api/agent-approval.js` with an `action=invite_admin` branch that takes `{email, full_name}`, calls Supabase admin `inviteUserByEmail`, then inserts into `admin_users` (no new Vercel function created — we are at the 12/12 ceiling). | A non-admin customer is rejected at `/admin/login.html`. Mandeep invites a teammate by email, teammate sets password via Supabase invite link, teammate signs in, sees the same dashboard. |

## 11. Testing strategy

- **Phases 1-3:** automated. CLI tests using `curl` against `/api/create-booking` + Supabase Admin client to create / verify / sign in a test user; assert RLS denies cross-user reads.
- **Phase 4:** manual on iOS device. New test agent registers → books → confirms appearance in My Bookings.
- **Phase 5:** manual. Mandeep signs in with the existing email + a freshly-reset password, confirms historic booking visible.
- Each phase follows TDD: write the failing test → write the code → green → commit.

## 12. Out of scope (explicit non-goals)

- Granular admin permissions (view-only, bookings-only, etc.) — v1.2 grants every admin in `admin_users` full access. Granular roles come in v1.3.
- Admin self-service password reset on `/admin/` — already exists via `forgot.html`, unchanged in v1.2.
- Stripe / deposit-at-booking — future v2.
- Push notifications — future v1.3+.
- Booking edit / cancel from mobile — future v1.3.
- Social / OAuth login (Google, Apple) — future v1.3+.
- Magic-link as a secondary sign-in option — removed in v1.2 (Supabase still uses it under the hood for password reset).

## 13. Risks

1. **Vercel function ceiling.** Audit shows we are already at **12/12** Hobby functions (memory said 11/12, reality says 12). Phase 6 therefore CANNOT add a new function. The admin-invite endpoint is multiplexed into `api/agent-approval.js` via a `?action=invite_admin` query param. Total stays at 12/12 after v1.2 ships.
2. **Backfill coverage.** Only bookings whose `agent_email` matches a real auth user get a `user_id`. Pre-v1.2 bookings made by unregistered agents stay orphaned, visible only in `/admin/`. Acceptable since there are ~10 test bookings in prod.
3. **Email deliverability.** Supabase SMTP is verified delivering to Mandeep's inbox. The `cron-keepalive` job currently runs every 3 days, which is the longest interval Supabase tolerates before auto-pausing the project. Recommend bumping to daily as part of Phase 1 to remove this risk.
4. **iOS App Store re-review.** Apple is currently re-reviewing the resubmitted iOS app. v1.2 changes are web-only — they reach iOS users on the next Vercel deploy with zero new Apple submission. No re-review needed.
